# planning.md — Provenance Guard

## Architecture

### Narrative

A piece of text enters the system at `POST /submit`. Before any detection runs, the
request passes through Flask-Limiter, which enforces per-IP rate limits and rejects
excess requests with a 429 before they touch the detection pipeline. If the request
is within limits, the app generates a `content_id` (UUID4), then runs the text through
two independent detection signals in sequence: first the LLM signal (Groq API), then
the stylometric signal (pure Python). The two raw scores are fed into the confidence
scorer, which produces a single float in [0, 1] — where values closer to 1 mean
"likely AI-generated." That score is passed to the label generator, which maps it to
one of three transparency label variants. Finally, a structured audit entry is written
(including both individual signal scores, the combined score, and the label), and the
API returns a JSON response to the caller.

An appeal enters at `POST /appeal`. The app looks up the original audit entry by
`content_id`, appends `appeal_reasoning` and `appeal_timestamp` to it, and sets the
entry's `status` to `under_review`. The audit log is the single source of truth —
the appeal is stored in the same record as the original decision, not separately. The
endpoint returns a confirmation. No automated re-classification happens; a human
reviewer would query `GET /log` to see all `under_review` entries.

## Detection Signals

### Signal 1 — LLM semantic assessment (Groq)

**What it measures:**  
Holistic semantic and stylistic coherence. LLMs trained on human writing have learned
that AI-generated text tends to exhibit particular patterns — overly balanced sentence
structures, transition phrases like "furthermore" and "it is important to note,"
consistent hedging, and an absence of the idiosyncratic tangents and register shifts
that characterise authentic human writing. The Groq model is asked to produce a
probability estimate, not a binary label.

**Why this property differs between human and AI writing:**  
Large language models optimise for fluency and plausibility, which produces text that
is locally coherent but globally uniform. Human writers make unexpected word choices,
shift register mid-paragraph, use personal references, make minor grammatical
deviations, and include details that feel incidental rather than purposeful. A
language model can detect these statistical fingerprints holistically in a way that
rule-based heuristics cannot.

**Output format:**  
A float in [0.0, 1.0]. 1.0 means the model believes the text is almost certainly
AI-generated. The model is prompted to return only a JSON object:
`{"ai_probability": 0.82}`. The app extracts this float as `llm_score`.

**Prompt:**
```
You are an AI content attribution system. Analyse the text below and estimate the
probability (0.0 to 1.0) that it was generated by an AI language model rather than
written by a human. 0.0 = certainly human-written, 1.0 = certainly AI-generated.

Return ONLY valid JSON in this format: {"ai_probability": <float>}
Do not include any explanation.

TEXT:
{text}
```

**Blind spots:**  
- Casual AI text (e.g. ChatGPT writing in a requested "informal" tone) may score lower
  than it should.
- Highly formal human writing (academic, legal) may score higher than it should.
- Short texts (under ~50 words) give the model too little signal; scores will be noisy.
- The model may be inconsistent across API calls for borderline texts.

### Signal 2 — Stylometric heuristics (pure Python)

**What it measures:**  
Statistical structural properties of the text that differ measurably between human and
AI writing. Specifically: vocabulary diversity (type-token ratio), sentence length
variance, and punctuation density. These are computed without any external libraries
beyond Python's standard `re` and `statistics` modules.

**Why these properties differ:**  
AI-generated text is statistically more uniform than human writing. It uses a more
consistent range of sentence lengths, a narrower effective vocabulary relative to text
length, and tends to under-use punctuation marks like em-dashes, colons, and
parentheses that human writers deploy for rhetorical effect. Human writing shows higher
variance on all three dimensions.

**Three sub-metrics computed:**

1. **Type-Token Ratio (TTR)** — `unique_words / total_words`. AI text tends toward a
   lower TTR (more word repetition). A TTR below ~0.55 on short-to-medium texts is a
   mild signal of AI generation. This metric degrades on long texts (TTR naturally
   falls as length increases) so it is capped at the first 200 tokens.

2. **Sentence length variance** — standard deviation of word counts per sentence. AI
   text tends to have low variance (sentences cluster in the 15–25 word range).
   Human text varies more widely. Variance below ~20 (low spread) is a signal of AI
   generation.

3. **Punctuation density** — count of `,;:—–()…` divided by word count. AI text uses
   commas consistently but under-uses em-dashes, parentheses, ellipses, and
   semicolons. A density below ~0.04 is a mild AI signal.

**Combining sub-metrics into `style_score`:**  
Each sub-metric is normalised to [0, 1] where 1 = "likely AI" and 0 = "likely human."
The three normalised sub-scores are averaged with equal weight:

```
style_score = mean([
    normalise_ttr(ttr),             # low TTR → high score
    normalise_variance(variance),   # low variance → high score
    normalise_punct(punct_density)  # low density → high score
])
```

Normalisation functions are linear clamps: each sub-metric maps its "clearly AI" end
to 1.0 and its "clearly human" end to 0.0, with values clamped to [0, 1].

**Output format:** A float in [0.0, 1.0] stored as `style_score`.

**Blind spots:**  
- Non-native English speakers writing formally may produce low TTR and low variance,
  scoring higher than warranted. This is a known false-positive risk.
- Poetry and highly structured prose (lists, numbered points) will inflate the score
  because structural regularity mimics AI uniformity.
- Very short texts (under ~30 words) produce unreliable sentence-length variance.
- Texts that use heavy quotation or repetition as a stylistic device (certain fiction
  genres) will artificially lower TTR.

## Combining Signals into a Confidence Score

**Weighting rationale:**  
The LLM signal is given more weight (60%) because it captures holistic, semantic
properties that stylometric heuristics miss. Stylometric heuristics are given 40%
because they are independent, deterministic, and can catch cases where AI text has
been lightly edited but the statistical fingerprint remains.

```
combined_score = (0.6 × llm_score) + (0.4 × style_score)
```

**Why not 50/50?**  
The stylometric signal has a known bias toward flagging formal human writing. Giving
it equal weight would push too many legitimate writers into the uncertain zone. A 60/40
weighting preserves its contribution while keeping the LLM signal dominant.

## Uncertainty Representation

**What does a score of 0.6 mean?**  
It means the system has a moderate lean toward AI generation, but both signals have
returned inconclusive results — neither is confidently flagging the text. At 0.6, the
label shown to the reader will explicitly acknowledge uncertainty and will not state
that the text is AI-generated. This is by design: on a creative platform, a false
positive (labelling a human writer's work as AI-generated) is more damaging than a
false negative.

**Thresholds:**

```
combined_score < 0.35   → attribution: "likely_human"
0.35 ≤ score < 0.65     → attribution: "uncertain"
combined_score ≥ 0.65   → attribution: "likely_ai"
```

The uncertain band is deliberately wide (30 percentage points) to avoid forcing
borderline texts into a binary. Anything less than a clear signal produces an
"uncertain" label.

**Asymmetry:**  
The threshold to reach "likely_ai" (0.65) is higher than the threshold to exit
"likely_human" (0.35). This means a text needs a stronger signal to be labelled AI
than to be labelled human — reflecting the higher cost of a false positive on this
platform.

## Transparency Label Variants

These are the exact strings returned in the `label` field of the API response and
displayed to readers on the platform. All three are implemented in a
`generate_label(combined_score)` function.

---

**Variant A — High-confidence AI (score ≥ 0.65)**

```
⚠ This content was likely generated with AI assistance.

Our system analysed this submission using two independent methods and found
strong indicators of AI-generated text. This label does not mean the work
has no value — it is provided so you can make an informed decision about
what you're reading.

If you are the creator and believe this is incorrect, you can submit an
appeal below.
```

---

**Variant B — Uncertain (0.35 ≤ score < 0.65)**

```
ℹ Attribution unclear

Our system found mixed signals in this submission. We could not reliably
determine whether this content was written by a human or generated with AI
assistance. No strong conclusion is drawn either way.

Creators can submit an appeal at any time to provide additional context.
```

---

**Variant C — High-confidence human (score < 0.35)**

```
✓ This content appears to be human-written.

Our system found no strong indicators of AI generation in this submission.
This is not a guarantee — detection tools are imperfect — but the signals
we measured are consistent with human authorship.
```

---

**Label design notes:**  
- Variant A explicitly acknowledges that being AI-assisted does not mean the work is
  worthless. This reduces the stigma of a flag and reduces adversarial appeals.
- Variant B does not say "we don't know" — it says we found mixed signals. This is more
  honest about what the system actually measured.
- Variant C hedges explicitly ("not a guarantee") to set accurate expectations and
  reduce disappointment if a misclassification is later discovered.
- All three mention the appeals path, or the absence of a hard verdict, to set user
  expectations correctly.

## Appeals Workflow

**Who can submit an appeal:**  
Any creator who has a `content_id` from a prior submission. No authentication is
required in this implementation (a real deployment would require creator authentication
to prevent third-party interference with another user's content status).

**What they provide:**  
- `content_id` — the UUID from the original submission response
- `creator_reasoning` — free text, their explanation (no minimum length enforced, but
  the field is required and must be non-empty)

**What the system does when an appeal is received:**  
1. Looks up the audit log entry for `content_id`. Returns 404 if not found.
2. Checks that `status` is not already `under_review` (returns 409 if re-submitted).
3. Updates the entry: sets `status = "under_review"`, adds `appeal_reasoning` and
   `appeal_timestamp`.
4. Returns a confirmation object:
   ```json
   {
     "content_id": "...",
     "status": "under_review",
     "message": "Your appeal has been received and will be reviewed by a human moderator."
   }
   ```

**What a human reviewer sees:**  
A query to `GET /log?status=under_review` returns all appealed entries. Each entry
contains the original `attribution`, both signal scores, the `combined_score`, the
transparency label shown at time of classification, the `creator_reasoning`, and the
`appeal_timestamp`. The reviewer has everything they need to make a decision without
re-running the pipeline.

**What automated re-classification is not implemented and why:**  
Automated re-classification on appeal would create a loop where creators iteratively
submit until they receive a "likely_human" label. Human review is a deliberate
friction point.

## Anticipated Edge Cases

**1. Non-native English speakers writing formally**  
A non-native speaker who writes in careful, grammatically correct, but somewhat formal
English will produce low sentence-length variance and moderate-to-low TTR — both of
which the stylometric signal interprets as AI-like. The LLM signal may also score
higher than warranted because formal non-native writing patterns can superficially
resemble AI output. This is the most serious false-positive risk in the system. The
wide uncertain band (0.35–0.65) is designed in part to catch this scenario; the
system should return "uncertain" rather than "likely_ai" for this writer.

**2. Lightly edited AI output**  
A creator who generates text with an LLM and then edits it — fixing passive voice,
adding personal anecdotes, varying sentence lengths — can meaningfully shift both
signals. The stylometric signal may drop if variance and TTR increase after editing.
The LLM signal may also drop if the edits introduce enough idiosyncrasy. The system
will likely return "uncertain" or, in heavily edited cases, "likely_human" — a true
false negative. This is an acceptable outcome given the asymmetric threshold design.

**3. Poetry and highly structured forms**  
Poems with short, regular line lengths, restricted vocabulary, and intentional
repetition will score high on the stylometric signal despite being deeply human.
Depending on the LLM signal, the combined score may push the text into the uncertain
range. This is a known limitation documented in the label system and should be
surfaced in the README.

**4. Very short submissions (under 40 words)**  
Both signals degrade on short text. TTR is inflated (few words, most unique), sentence
variance is unreliable (few sentences), and the LLM has little signal to work with.
The system should ideally refuse or warn on texts below a minimum length. Minimum
length enforcement (`len(text.split()) < 40`) should be added as a validation step
that returns a 400 with a clear error message.

## Rate Limiting

**Chosen limits:**  
- 10 requests per minute per IP
- 100 requests per day per IP

**Reasoning:**  
A real creator submitting their own work would realistically submit a few pieces per
session — perhaps 5–10 in a sitting, never 10 within 60 seconds. The per-minute limit
of 10 is permissive enough to allow legitimate batch testing but low enough to stop a
naive flood attack. The daily cap of 100 prevents a slow, distributed drip attack that
would evade the per-minute limit while still consuming Groq API credits.

The Groq free tier has its own rate limits; the Flask-Limiter cap prevents the app
from making 100+ Groq API calls in a minute even if one IP hammers the endpoint.
